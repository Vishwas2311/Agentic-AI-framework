FAQ

          
Common questions about Genesis

        

      

      

        

          
          
A migration is a single run of nocode2procode migrate that converts your no-code evidence into a production-ready application. One migration reads inputs, resolves source truth, generates frontend + backend + database + tests + deployment artifacts, runs QA, and saves verified learning. Each migration produces a full, reviewable output — not a partial scaffold.

        

        

          
          
Genesis supports PowerApps exports (.pa.yaml, packages), website references (URLs or static files), BRD documents (PDF, DOCX, Markdown), runtime videos (user walkthroughs), database schemas (SQL, CSV, ERD), API configs (OpenAPI, YAML, JSON), and screenshots. Every source is treated as structured evidence, not just raw input. New extractors are added regularly.

        

        

          
          
The source_truth_agent compares all extracted evidence — PowerApps AST, BRD, website structure, runtime video, and database schema — looking for contradictions. When sources disagree, it assigns confidence scores, records the conflict, and either resolves it automatically (when confidence is high) or flags it for human review. Nothing moves to generation until source truth is resolved.

Genesis generates a complete React or Next.js frontend, FastAPI or Node.js backend, database schema with migrations, Playwright UI tests, API tests, a security review report, Docker configuration, and GitHub Actions CI. The QA repair loop runs automated checks and fixes failures before you ever review the output. "Production-ready" means it passes all quality gates, not just that it compiles.

Specify React + FastAPI, or Next.js + Tailwind + shadcn/ui for a premium portal — it applies the right profile through UI/UX Pro Max. The stack decision is recorded in stack_decision_report.json before generation begins, so there are no silent choices. You know what is being generated and why before a file is written.

        

        

          
          
Verified memory is Genesis's learning layer. After each successful migration, it saves a verified_memory_packet.json containing reusable patterns — naming conventions, screen structures, workflow shapes, approval flows — that passed build, tests, and human review. On the next migration, Genesis retrieves relevant patterns before planning begins, so output starts at a higher baseline. The longer you use Genesis, the fewer corrections you need.
